오늘 한 일

keyframes와 animation 속성을 사용해 간단한 애니메이션을 만들어보고 로그인 섹션의 마크업 구조와 적용할 스타일에 대해 다양하게 고민을 하고 만들어 보았다. 또한 BFC와 flow, 그리고 접근성에 관련된 여러 레퍼런스와 아티클을 읽어 보았다.

느낀점

그동안 마크업과 css는 전혀 어려울게 없고 그저 다양한 속성들만 더 공부하면 되겠다 생각했었는데 작은 섹션 하나를 구축할 때에도 접근성과 성능 등을 고려해 구조를 설계하고 스타일을 입히는 방법을 배우면서 html과 css가 단순한 영역이 아니라는 것을 깨달았다. 또한 성능문제까지 고려해야하니 설계부터 많은 공을 들여야한다. 코드부터 치기 전에 설계도를 그리는 습관을 들여야겠다. 접근성에대한 이해를 돕기 위해 관련 레퍼런스를 읽으면서 접근성을 고려한 웹은 단순히 다양한 사용자에게만 좋은 것이 아니라 많은 사용자를 확보할 수 있고 성능과 속도면에서도 좋다고 한다. 오 1석2조 1타쌍피.
수업 때마다 툭툭 위어나오는 BFC라는 것이 도통 이해가 안가서 며칠동안 이거에대한 참고자료만 찾아봤던 것 같다. 완벽하게는 아니더라도 어느정도 이해가 되고나니 그동안 수업시간에 따라가지 못했던 개념들이 많이 해소가 되었다. 드디어 여유가 좀 생겼다. 이렇게 블로그에 글도 올리고. 역시 개념이 이렇게나 중요하다.
인터랙티브 웹을 정말 좋아하는데 그래서 오늘 애니메이션 실습이 너무 재밌었다. 직접 여러 예제를 따라해 보면서 더욱 다양한 효과를 내는 애니메이션을 만들어보고 싶은 마음이 생겼다. 시간이 난다면 간단한 효과부터 하나씩 만들어보려고 한다. 시간아 허락해다오.

웹 접근성

웹 접근성

웹에 액세스(접근)하는 사람과 방법에 관계없이 모든 사용자가 신체적, 환경적 조건에 관계없이 웹을 동일하게 접근할 수 있도록 제작하는 방법. 단순히 장애인에 대한 것이라고 생각하지만, 정상적인 웹 사용이 가능한 일반 사용자부터, 고령자, 신체적 장애로 특수한 환경에서 접속해야 하는 사용자 등이 어떠한 브라우저를 선택해 사용하던 어떠한 기기로 웹을 이용하던 어떠한 도구로 접근을 하던 네트워크 연결이 느린 환경이던 빠른 환경이던 정보제공이 동일하게 이루어져야함. 이런 사용자와 환경들을 고려해서 대체 텍스트를 제공하고 논리적인 아웃라인을 만들며 탭키로 모든 링크와 이미지 등에 논리적인 순서대로 접근이 가능하게 만드는 것과 크로스 브라우징은 기본으로 깔고 웹을 만들어야함. 접근성이 높은 웹을 만들기 위해서는 자신의 컴퓨터와 웹 사용 방식을 넘어서 생각하고 다른 사람이 웹을 사용하는 방식에 대해서 많은 생각을 해야함. 접근성을 향상시키는 시맨틱한 마크업은 SEO(검색 최적화)를 향상시켜 사이트를 더 쉽게 찾을 수 있도록 만들어주는 등 접근성을 고려해 만든 사이트는 많은 이용자를 확보할 수 있고 속도와 성능도 빠름.
접근성 문제가있는 기존 웹 사이트를 개조하는데에는 많은 비용이 들기 때문에 프로젝트 시작부터 고려해야함.
Firefox 접근성 검사기는 접근성 문제를 확인하는 데 매우 유용한 도구.

참조 : https://developer.mozilla.org/ko/docs/Learn/accessibility/What_is_accessibility

CSS tip, 요소 정렬 방법

CSS tip

CSS는 적은 코드로 크로스 브라우징, 유지보수를 용이하게 할 수 있도록 여러가지를 고려해서 작업해야함.

  • 폰트 단위
    em단위를 많이 사용하지만 rem단위를 사용하면 모든 폰트 사이즈를 통일되게 키우거나 작아지게 만들고 싶을때 root사이즈 하나만 변경하면 되므로 용이하게 사용 가능.
  • default box사이즈 설정
    요소에 정해놓은 사이즈가 padding이나 border값에 영향받지 않도록 하기 위해 사용. 예로 모바일의 경우 width: 100%로 설계해야하지만 양쪽에 여백값을 주어 콘텐츠를 읽기 쉽게 만들어주어야함. 이 때 내부에 여백을 주면 너비가 100%를 넘어가게 되므로 기본 박스 사이즈를 설정해놓으면 영향받지않고 사용할 수 있음.
    브라우저별 가상요소 선택자의 설정이 다르게 되어있을 수 있으므로 ::before, ::after가상요소까지 선택해서 설정해주는 것이 좋음. 모든 요소를 선택해서 사용하지 않는 모든 요소의 박스사이즈를 바꾸는 것 보다 안에 있는 모든 요소만을 선택해 사용 하는 것이 성능면에서 좋음.
  • 요소들의 높이는 요소의 콘텐츠나 자식 요소의 높이만큼 자연스레 가질 수 있도록 되도록이면 설정하지 않는 것이 좋음. 유연하게 사용 가능.

BOX MODEL

  • contents box 동작방식 : padding, border, margin을 포함시킴.
    margin: auto; 브라우저가 차지하는 영역에서 콘텐츠의 너비를 뺀 나머지의 값을 2분할 하여 왼/오른쪽에 margin을 줌. -> 요소를 가운데 정렬시킬 때 사용.
  • border box 동작방식 : margin를 뺀 나머지 padding, border를 포함시킴.

요소 정렬(배치)방법

  • inline-block
    정렬시킬 요소의 display값을 inline-block으로 설정해 크기 설정이 가능해지는 등 블록 성격을 가지면서 인라인 방식으로 정렬.
  • float
    정렬시킬 요소의 속성으로 float을 넣어주고 값으로 left/right을 넣어 정렬시킬 위치를 줌(float:right;사용 시 우측에 마크업 순서대로 붙기 때문에 화면에는 역정렬된걸로 보임). display: flex;를 사용할 수 없는 구형 브라우저를 커버할 때 사용.
    float을 주면 이 때 자신이 포함된 부모의 크기만큼 가상의 linebox를 만드는데 속성 값에 따라 그 linebox의 왼/오른쪽으로 배치가됨. float 요소의 크기가 linebox밖을 벗어나게 되면 밑으로 떨어져 다시 linebox의 위로 붙어 왼/오른쪽으로 배치됨. 이 때 요소들의 높이가 다를경우 제일 높은 박스를 기준으로 linebox가 그려지기 때문에 높이가 다른 요소가 정렬됐을 시 그 공간에 갭이 생길 수 있음.
    float된 요소가 인라인요소를 만나 겹쳐졌을 때 인라인 요소의 콘텐츠가 겹친 부분 밖으로 밀려나게됨.
    float을 준 요소가 부유를 하면서 부모는 자식의 높이를 잃어버게됨. 부모가 자연스레 자식의 높이를 가지게 하려면 float을 해제시켜줘야함. 해제 하지 않으면 뒤따라오던 콘텐트와 엉켜버림.
    해결방법 :
    • html에서 class="clearfix"를 가진 빈 요소를 float준 요소의 마지막 형제 요소로 하나 추가하고 css에서 .clearfix {content: ""; display: block; clear: both;} 설정해서 해제.
      이 경우 부모에 높이를 줄 수는 있지만 html에 구조상 필요없는 요소를 추가해야하고 유연하게 사용하기 어려워 좋지 않은 방법.
    • 부모 속성에 height를 주지 않은 상태에서 overflow: hidden; 속성 추가. 이 때 부모가 자식의 크기를 계산하면서(자식 요소의 크기를 알아야 overflow 컨트롤을 할 수 있게 되므로) 하면서 계산된 자식의 height가 잡게되어 그만큼의 높이가 자연스레 잡히게 됨. BFC(Block Formatting Context). 오버플로우 히든 방식역시 유연한 콘텐츠를 사용하기 어려움.
    • css에서 .clearfix::after {content: ""; display: block; clear: both;} 가상요소를 만들어 설정하고 float된 요소의 부모 요소에 .clearfix 클래스를 추가해서 해제.
  • flex
    정렬시킬 요소의 부모에 display: flex;를 주고 flex 속성들로 정렬 방식을 지정. 모던브라우저 커버할 때 사용.

HTML 기본 구조와 레이아웃

HTML 기본 구조

  • <!DOCTYPE html>
    선언된 페이지의 HTML 버전이 무엇인지를 웹 브라우저에 알려주는 역할을 하는 선언문.
    HTML 4.01에서 DOCTYPE 선언은 SGML을 기반으로 하기 때문에 DTD를 참조해야 함. DTD는 브라우저가 콘텐츠를 정확하게 표현하도록 마크업 언어에 대한 규칙을 명시.
    하지만 HTML5는 SGML을 기반으로 하지 않기 때문에 DTD를 참조할 필요가 없음.

  • <html lang="ko-KR">
    HTML 문서의 루트 요소(root element)를 정의할 때 사용. DOCTYPE 선언을 제외한 모든 다른 HTML 요소를 포함하기 위한 컨테이너이며, 브라우저에게 해당 문서가 HTML 문서임을 알려주는 역할을 함.
    lang속성으로 해당 문서의 언어를 지정해줌. 값을 ko를 넣어도 되지만 ko-KR를 넣어 남한에서 사용하는 한국어를 표시.

  • <head></head>
    해당 문서에 대한 정보인 메타데이터(metadata)의 집합을 정의할 때 사용. <title>, <style>, <base>, <link>, <meta>, <script>, <noscript> 같은 요소들이 포함됨.

  • <meta charset="UTF-8">
    문자의 인코딩방식 선언. utf-8은 다국어 지원. 예전엔 한글 완성형인 EUC-KR이 많이 쓰였음.

  • <title>타이틀</title>
    해당 문서의 제목(title)을 정의할 때 사용. 검색과 연관성있는 영역이기 때문에 핵심 키워드를 포함해야함. SEO(검색 최적화)O를 고려해서 작성하지 않으면 접근성을 포기하게 됨.

  • <link rel="stylesheet" href="경로">
    해당 문서와 외부 소스(external resource) 사이의 관계를 정의할 때 사용. 빈 태그이며 href속성을 통해 css파일을 첨부.

  • <body></body>
    해당 HTML 문서의 텍스트, 하이퍼링크, 이미지, 리스트 등과 같은 모든 콘텐츠를 포함하는 영역을 정의할 때 사용.


[참조](http://tcpschool.com/html-tags/)


레이아웃

  • 3단 레이아웃
    header, main, footer로 나누고 보통 visual과 slogan섹션은 따로 분류.

  • Emmet 기능
    main.main>div.group.group${group$}*3 이런식으로 태그명, 클래스명, 자식요소 등을 한줄로 표기해 탭키나 엔터키를 사용하면 한 번에 해당 요소들이 구조대로 짜여짐. $는 번호를, {}는 노드 영역에 들어갈 내용이 됨.

  • <div class="skip-nav"><a href="#">본문 바로가기</a></div>
    접근성을 고려해 웹에서 키보드의 탭키로 콘텐츠 이동 시 반복되는 내용(내비)을 스킵. 가장 첫머리에 삽입.

BEM (CSS 방법론)

CSS 방법론

CSS 구조를 개선시켜 코드를 읽기 쉽고 효율적으로 만들어 개발 속도를 높이고 디버깅 및 레거시 코드의 새로운 기능 구현을 크게 향상시키기 위해 사용되는 CSS 네이밍 규칙.
OOCSS, SMACSS, BEM 등이 표준으로 많이 사용되고 있음.

BEM(Block Element Modifier)

  • BEM 방법론은 id속성은 사용하지 않고 오직 class만 부여함.
  • 두 개 이상의 단어를 조합할 땐 -(dash)로 이어주고 블록과 요소의 관계를 나타낼 때엔 _(underscore)를 두번 사용해 이어줌.
    예시)
    .block__element‐‐modifier { property: value; }
  • class명을 지을 때의 목적은 그 요소를 반복하여 재사용할 수 있게 하기 위함. 모든 class명은 구체적이고 명료하고 직관적으로 무엇을 나타내는지 누구나 이해할 수 있게 지어주는 것이 좋음.
  • 각 html 태그에는 오직 class명 하나만 사용해야함. 한 태그 안에 두 개 이상 class 사용하지 않음.

Block(전체를 감싸고 있는 블록요소)

블록(block)은 요소들를 담고 있는 컨테이너로 큰 맥락의 구조. header, main, aside, footer등.
블록 요소는 class의 어근을 형성하고 항상 맨 앞에 위치하도록 함.

Element(요소)

요소(Element)는 블록이 포함하고 있는 한 조각. 블록은 전체를 말하고 요소들은 블록 안에 있는 조각들을 말함. 각 요소는 두 개의 _(underscore)로 연결하여 블록 다음에 위치시킴.

Modifiers

블록이나 요소의 외관, 상태를 변화시킬 때 사용. 요소의 스타일이 같다면 다른 영역(다른 블록 안의 요소)이라 할지라도 새로운 클래스를 정의하지 않고 뒤에 -(dash)를 두 개 추가하고 modifier를 표시함. 절대적으로 필요하지 않는 이상, 추가적인 class를 만들거나 똑같은 스타일을 반복하지 않음.
예시) .header__navigation‐‐secondary {…}

웹페이지의 골격을 만드는 부분으로 체계적인 구조설계가 중요. 나만의 convention(코딩 가이드)를 만들어 기준을 정해 만들어야함.

  • 구조
    웹페이지는 기본적으로 헤더(내비게이션 포함), 콘텐츠, 푸터로 짜여진 3단구성과 헤더에서 내비게이션이 불리된 4단구성이 있음.
  • 논리적인 순서
    마크업 순서는 눈에 보여지는 디자인 베이스 순서가 아닌 책의 목차와 비슷한 맥락으로, (CSS 제거했을 때)제작된 웹 문서의 개요를 쉽게 알 수 있게 논리적인 종속과 순서로 이루어져 있어야함. 키보드의 탭키로 콘텐츠 이동시 순서대로 접근할 수 있게 만들어야함.
  • 시맨틱 마크업
    해당 콘텐츠를 기계가 이해하기 쉽게 의미를 넣어 알려주는 것. 각각의 의미와 사용 목적이 있는 태그를 적절하게 사용해야함. 예로 예전에 레이아웃 짤 때 많이 사용했던 은 데이터를 보여주기 위한 태그이므로 레이아웃을 잡는 용도로 적합하지 않음.
  • 네이밍
    각 블록과 요소에 id나 class속성을 추가해 해당 요소를 직접적으로 선택 가능하게 만들어주는 것. 누구나 이름만 보고도 요소의 위치나 기능 등을 이해할 수 있게 만들어야함. 두 개 이상의 단어를 연결하는 네이밍 기법에는 KC(kebab-case), CC(camelCase), PC(PascalCase), SC(snake_case)기법 등을 사용. 요즘은 class 기반으로 네임을 주고 id속성은 사용하지 않음. 트렌드에 맞는 네이밍이 필요. 예시) log in -> sign in, join -> sign up
    CSS의 규모가 확장되고 프로젝트의 규모가 커질수록 관리하기 어려운 CSS를 관리하기 수월하게 하고자 쉬운 유지보수, 코드의 재사용, 확장 가능, 직관적인 네이밍을 지향한 설계방법인 CSS 방법론(SMACSS, BEM, OOCSS)이 있음 BEM(CSS 방법론)

    웹폰트

    운영체제마다 기본 폰트가 다르기 떄문에 웹폰트를 사용하는 것이 좋음. 용량이 큰 경우 플래쉬 현상(깜빡거림)이 있을 수 있음. 보통 구글의 Noto Sans, 어도비의 Spoqa Han Sans를 많이 사용. 웹폰트는 크로스 브라우징 이슈가 있을 수 있어 각 브라우저에 맞는 확장자(ttf:애플과 MS가 개발한 글꼴 표준, oft:확장 가능한 컴퓨터 글꼴, woff/woff2:웹페이지에서 사용하기 위한 글꼴, eot: 웹에 포함 된 글꼴을 사용하도록 설계)를 가진 웹폰트를 모두 넣어줘야함.

    Git/Github 사용법

    Git/Github 사용법

    Shell 명령어

    • ls : (list segment) 현재 디렉토리에 있는 디렉토리/파일 목록 보기
    • ls -a : 숨김 디렉토리/파일 포함한 목록 보기
    • ls -l : 각 디렉토리/파일의 상세 정보 보기
    • ls -al : 숨김 디렉토리/파일 포함해 상세정보 보기
    • pwd : (print work directory) 현재 디렉토리의 절대경로를 보여줌
    • clear : 터미널의 명령을 모두 지움
    • cd 디렉토리명 : (change directory) 해당 디렉토리로 이동
      cd .. : 상위 디렉토리로 이동
      cd ../디렉토리명 : 상위 디렉토리에 있는 해당 디렉토리로 이동
    • mkdir 디렉토리이름 : (make directory) 현재 디렉토리에 새로운 디렉토리 생성(html, css, 이미지파일 등 정적인 파일들을 넣는 폴더명은 static이나 public을 사용)
    • touch 파일이름 : 현재 디렉토리에 새로운 파일 생성(pptx, dn, hwp 등의 확장자 파일들은 어플리케이션을 통해 생성되야하기떄문에 (텍스트기반파일) shell 로 만들어도 동작하지 않음. pre conpile이 되어 있어야 함)
    • mv 파일명 디렉토리명 : (move) 현재 디렉토리에서 해당 파일을 해당 디렉토리로 이동
      mv ../파일명 . : 상위 디렉토리에 있는 파일을 현재 디렉토리로 이동
    • mv 기존파일명 바꿀파일명 : 기존 파일명을 해당 파일명으로 변경
    • cp 파일명 디렉토리명 : 해당 파일을 해당 디렉토리에 복사
      cp 파일명 .. : 해당 파일을 상위 디렉토리에 복사
    • rm 파일명 : 해당 파일 삭제
      rm 공통이름* , rm *공통이름 : 공통이름을 가진 해당 파일 모두 지움
    • rm -r 디렉토리명 : 디렉토리 삭제(디렉토리 안의 모든 오브젝트까지 삭제)
    • rm -rf 디렉토리명 : 모든 설정 무시하고 디렉토리 강제 삭제
      rm -rf / : 자기 자신(하드디스크 전체)을 지움. 절대 사용 금지!!!!!
    • code 파일명 : 해당 파일을 VScode로 열어줌
    • vi 파일명 : 해당 파일을 vim으로 열어줌
    • open 파일명 : 해당 파일을 열어줌
    • chmod [옵션] 8진수 디렉토리명/파일명 : 해당 디렉토리/파일을 해당 권한으로 재정의
      MacOS와 Linux에서는 ‘ls -l’를 했을 때 해당 디렉토리/파일의 권한이 제일 앞에 알파벳 열자리로 보여짐(윈도우는 안됨). 
    1
    2
    drwxr-xr-x 1 pc 197609 0 10월 16 11:43 fds/
    -rw-r--r-- 1 pc 197609 0 10월 16 11:44 readme.md
    알파벳의 첫번째 자리는 해당 오브젝트가 디렉토리인지 파일인지를 보여줌. d는 디렉토리, -는 없음을 뜻함(디렉토리가 아니라는 뜻).
    나머지 아홉자리의 알파벳은 세자리씩 끊어서 처음 세자리는 나, 두번째 세자리는 나를 포함한 그룹, 마지막 세자리는 게스트를 뜻함. r은 읽기, w는 쓰기/수정, x는 실행을 뜻함. 해당 알파벳이 있다면 권한이 있고 -라면 권한이 없다는 뜻.
    chmod 명령어로 권한 변경 가능.

Git 명령어

  • git status : git 상태를 보여줌
  • git status -uall : 디테일한 경로까지 펼쳐서 보여줌(상단 디렉토리에서 사용하는게 좋음)
  • git log –oneline : git의 현재 상태를 한 줄로 보여줌
  • git init : 현재 디렉토리에서 현 시점부터 git을 쓰겠다고 초기화(꼭 git을 사용할 디렉토리 안에서 명령해야함)
  • git remote : 원격 저장소를 보여줌
  • git remote add origin URL주소 : 현재 디렉토리에 해당 주소의 remote 저장소를 추가(보통 원격저장소의 이름이 자동적으로 ‘origin’으로 되기 때문에 ‘origin’을 사용)
  • git remote -v :
    origin remoteURL (fetch) -> 디렉토리를 불러올 remote를 보여줌
    origin remoteURL (push) -> 디렉토리/파일을 보낼 remote를 보여줌
  • git remote get-url origin : remote 저장소의 URL을 보여줌
  • git clone URL주소 : 해당 주소의 remote 저장소 전체를 현재 디렉토리에 복사(현재 디렉토리에 해당 remote 저장소 전체 폴더가 생성됨)
  • git clone URL주소 . : 해당 주소의 remote 저장소의 디렉토리/파일들만 해당 디렉토리에 복사
  • git add 파일명 : 해당 파일을 stage에 올림
  • git commit : stage에 있는 파일들을 커밋하고 커밋 메세지를 쓸 수 있는 환경으로 바뀜
  • git commit -m “커밋 설명” : 해당 메세지로 커밋함
  • git commit -am : add과정 생략하고 바로 커밋함
  • git push origin(remote명) master(브랜치명) : 로컬 저장소에 있는 현재 커밋을 해당 브랜치에 올림
  • git push -u origin master : 처음 remote 디렉토리에 push할 때 사용


git commit 메세지 관례(Conventional Commits)

  • feat: 메세지타이틀 → features : 기능 개발
  • docs: 메세지타이틀 → documentation : 설명 문서
  • conf: 메세지타이틀 → configuratioin : 프로그램이 돌기 위한 환경구성 파일 작성
  • fix: 메세지타이틀 → bug-fix : 버그 수정
  • ci: 메세지타이틀 → continuous integration
  • test: 메세지타이틀 → test codes : 테스트 코드
  • hot-fix: 메세지타이틀 → hot-fix : 급한 수정
  • refactor: 메세지타이틀 → refactoring : 동작은 변경없지만 메소드 등을 완전히 변경
